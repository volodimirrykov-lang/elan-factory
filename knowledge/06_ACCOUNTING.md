# 06 — Accounting

Как ведётся учёт, кто какой кусок делает, где какие данные.

## Системы

| Система | Для чего | Кто ведёт | Доступ |
|---|---|---|---|
| **1С** | ELAN KIMYA (TR) основной учёт | Turkish бухгалтер | не автоматизировано |
| **Турецкая бух. программа** | TR налоговая отчётность (KDV, муҳтасар) | Турецкий бухгалтер | не автоматизировано |
| **Excel (RAPORLAR)** | Управленческий учёт, monthly close | Украинский бухгалтер (сводит) | xlsx через Telegram |
| **УЧЕТ НАЛИЧНЫХ АГ.xlsx** | Личная касса Анны Кравченко, "другое направление" | АГ лично | xlsx |
| **Зарплата MONTH.xlsx** | Payroll расчёт + export bonus | Украинский бухгалтер | xlsx |
| **Расходы на стройку.xlsx** | CAPEX tracker (ремонт + оборудование) | Володимир + УА бухгалтер | xlsx |

## Основа учёта

**Cash basis в основном**, с элементами accrual:
- Оплаты записываются по факту движения денег
- Инвойсы (дебеторская задолженность) отдельно в RAPORLAR
- Склад (inventory) — периодически (раз в месяц) инвентаризация

## Monthly close workflow

Каждый месяц ~5-10 числа следующего месяца:

1. **TR бухгалтер** закрывает 1С + налоговую программу
2. **УА бухгалтер** собирает данные из 1С + платёжных выписок банков + кассы АГ
3. **УА бухгалтер** формирует **RAPORLAR MONTH.xlsx** (все 4 листа: ИТОГ / ОПЛАТЫ / ПОСТУПЛЕНИЯ / ПРОДАЖИ)
4. **УА бухгалтер** отдельно формирует **Зарплата MONTH.xlsx** (payroll + premia)
5. **АГ лично** ведёт **УЧЕТ НАЛИЧНЫХ АГ** — один xlsx с 5+ листами (помесячно)
6. **Володимир** получает все xlsx через Telegram → кидает в `_raw/` → прогоняет скрипты → commit

## FX-политика (текущая, требует формализации)

- **Первичка**: TL (турецкая лира)
- **Ежемесячная сводка**: EUR
- **Cross-rate**: фиксируется один раз в месяц на дату закрытия
  - Март 2026: TL/EUR = **51.10**
  - Февраль 2026: TL/EUR = ~51 (другой курс)
- **USD** используется только для:
  - Комментариев к оценке ($1B target)
  - Переводов в Украину (через Wise/etc)
  - Касса АГ итоги (пересчёт всех приходов в USD)

⚠️ **Проблема**: курс не зафиксирован в коде/данных. Разные месяцы пересчитаны по разным курсам. Для консолидации нужна monthly close rate table.

## Сверка (текущая, ручная)

Раз в месяц:
1. Сумма клиентов ≈ ПРОДАЖИ (в ИТОГ листе RAPORLAR) — в пределах 1%
2. Касса AG: начальный + приход - расход = конечный (математический check)
3. Банк-балансы: начало + приход - расход = конец (9 счетов)
4. Инвентаризация склада: физический пересчёт vs xlsx

Нет автоматизации. В Level 1 добавить `scripts/validate.py`.

## Что НЕ автоматизировано (главные gaps)

1. **Intercompany elimination** — ФОП UA покупает у TR, продаёт внешним. В RAPORLAR это просто "выручка от ФОП". Но для консолидации нужно убрать
2. **Accrual adjustments** — нет механизма учёта доходов/расходов которые начислены но не оплачены
3. **Amortization / Depreciation** — CAPEX просто списан в opex. Для P&L должна быть амортизация
4. **Inventory movements** — склад — snapshot на конец месяца. Нет движений "принял — выдал в производство"
5. **Related-party flagging** — нет автоматической маркировки related vs external в отчётах
6. **Tax provision** — turkish corporate tax (20%) нигде не отражён
7. **Deferred revenue** — если клиент оплатил наперёд, это не учитывается
